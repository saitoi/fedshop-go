# fedshop-go User Guide

This guide explains how to build, run, benchmark, and debug `fedshop-go`, the
standalone Go federated SPARQL engine used by the local `fedshop-py` benchmark
pipeline.

`fedshop-go` is intentionally narrow. It targets the twelve FedShop BSBM query
templates and emits FedShop-compatible artifacts: result rows, source selection,
query plan, and engine statistics.

## Repository layout

```text
go-engine/
  cmd/fedshop-go/main.go           # CLI entry point
  internal/                        # parser, source selection, planner, executor, artifacts
  target/config/config_batch*.ttl  # generated federation configs
  README.md                        # short command reference
  USER_GUIDE.md                    # this guide

fedshop-py/
  config/config_small.yaml         # default small benchmark config
  queries/q01.sparql ... q12       # query templates
  src/fedshop/cli.py               # fedshop pipeline CLI
  src/fedshop/engines/fedshop_go.py # Python adapter for fedshop-go
  benchmark/                       # generated queries and evaluation outputs
```

## Prerequisites

Install:

- Go 1.23 or newer.
- Python 3.12 or newer for `fedshop-py`.
- `uv` for Python dependency and command execution.
- Docker if you want to generate data, ingest into Virtuoso, run the proxy, or
  execute benchmark runs against local services.

Run commands from the repository root unless a section explicitly changes
directory.

## Build and test the engine

```bash
cd go-engine
GOCACHE=$PWD/.gocache go test ./...
GOCACHE=$PWD/.gocache go build -o fedshop-go ./cmd/fedshop-go
```

Use the local `.gocache` so builds stay inside the workspace.

For race checks:

```bash
cd go-engine
GOCACHE=$PWD/.gocache go test -race ./...
```

Show the installed command surface:

```bash
cd go-engine
./fedshop-go --help
./fedshop-go version
```

## How fedshop-go works

A normal query run has five stages:

1. Parse the SPARQL SELECT query.
2. Read a Turtle federation config containing `sd:Service` entries.
3. Select candidate endpoints for each triple pattern.
4. Plan triple-pattern evaluation order.
5. Execute remote SPARQL requests and write benchmark artifacts.

The current implementation supports the SPARQL shape used by the FedShop
templates: basic graph patterns, FILTER, OPTIONAL, UNION, DISTINCT, ORDER BY,
OFFSET, and LIMIT. It does not aim to be a general-purpose SPARQL engine.

## Federation config format

`fedshop-go` reads the same service-description form generated by the Python
adapters:

```turtle
@prefix sd: <http://www.w3.org/ns/sparql-service-description#> .
@prefix fedx: <http://rdf4j.org/config/federation#> .

<http://www.vendor0.fr/> a sd:Service ;
    fedx:store "SPARQLEndpoint";
    sd:endpoint "http://localhost:8890/sparql?default-graph-uri=http%3A%2F%2Fwww.vendor0.fr%2F";
    fedx:supportsASKQueries true .
```

The subject IRI is the federation member graph IRI. The `sd:endpoint` value is
the SPARQL endpoint used by the engine.

For Virtuoso-backed FedShop runs, every endpoint must be graph-scoped, usually
through `default-graph-uri`. If all endpoints point at the same unscoped
`/sparql` URL, every endpoint exposes the same graph and source selection
becomes invalid.

## Run a standalone query

Build the binary first:

```bash
cd go-engine
GOCACHE=$PWD/.gocache go build -o fedshop-go ./cmd/fedshop-go
```

Then run one query:

```bash
cd go-engine
./fedshop-go query \
  --config target/config/config_batch0.ttl \
  --query ../fedshop-py/benchmark/generation/q01/instance_0/injected.sparql \
  --out-result /tmp/fedshop-go-results.csv \
  --out-source-selection /tmp/fedshop-go-source-selection.csv \
  --query-plan /tmp/fedshop-go-plan.txt \
  --stats /tmp/fedshop-go-stats.json \
  --selector ask \
  --cache memory \
  --join bind \
  --planner source-count \
  --exclusive-groups \
  --max-concurrency 4 \
  --bind-batch-size 20 \
  --retry-count 2 \
  --timeout 120s
```

Use `--http-proxy http://localhost:5555` only when you intentionally want the
Go HTTP client to route all endpoint traffic through the FedShop proxy as a
standard HTTP proxy. The default `fedshop-py` adapter does not pass this flag;
it writes endpoint URLs that are directly reachable from the host process.

## Query command options

Required flags:

| Flag | Meaning |
|---|---|
| `--config` | Turtle federation config with `sd:Service` endpoint entries. |
| `--query` | SPARQL SELECT query file. |
| `--out-result` | CSV result output path. |
| `--out-source-selection` | CSV source-selection output path. |
| `--query-plan` | Human-readable query-plan output path. |
| `--stats` | JSON engine-stats output path. |

Execution flags:

| Flag | Default | Values | Meaning |
|---|---:|---|---|
| `--timeout` | `60s` | Go duration | Whole query deadline and per-request timeout. |
| `--selector` | `ask` | `broadcast`, `ask`, `summary` | Source-selection strategy. |
| `--summary` | empty | path | Required for `--selector summary`. |
| `--cache` | `memory` | `memory`, `off` | Query-local ASK cache. |
| `--join` | `hash` | `hash`, `bind` | Local hash join or batched VALUES bind join. |
| `--planner` | `source-count` | `source-count`, `cost` | Greedy source-count order or summary-cardinality order. |
| `--failure-policy` | `strict` | `strict`, `partial` | Abort on endpoint error or emit partial execution stats. |
| `--http-proxy` | empty | URL | Explicit HTTP proxy for endpoint traffic. |
| `--retry-count` | `2` | integer | Retries for transient transport errors and HTTP 502/503/504. |
| `--max-concurrency` | `16` | integer | Maximum concurrent endpoint work. |
| `--bind-batch-size` | `20` | integer | Number of VALUES rows per bound request. |
| `--exclusive-groups` | `false` | boolean flag | Combine patterns assigned to one endpoint. |
| `--noexec` | `false` | boolean flag | Plan and write artifacts without SELECT execution. |

Use `--noexec` to inspect parsing, source selection, and plans without paying
for remote result materialization.

## Output artifacts

`fedshop-go query` writes four files.

### Results CSV

`--out-result` is a CSV file. The header is the SELECT projection order. Each
row contains lexical values only.

Example:

```csv
product,label
http://example/product1,Product 1
```

### Source-selection CSV

`--out-source-selection` is a CSV file with:

- `triple`: canonical triple-pattern text.
- `source_selection`: JSON array of selected endpoint IDs.

Example:

```csv
triple,source_selection
?s <http://example/p> ?o,"[\"http_www_vendor0_fr\"]"
```

Endpoint IDs are derived from graph IRIs by replacing non-identifier characters
with underscores and trimming leading or trailing underscores.

### Query plan

`--query-plan` is a human-readable debugging file. It records selector,
planner, join strategy, triple order, and one line per triple pattern. Statement
kinds are:

- `EmptyStatementPattern`: no selected source.
- `ExclusiveStatement`: exactly one selected source.
- `StatementSourcePattern`: multiple selected sources.

### Engine stats JSON

`--stats` is consumed by `fedshop-py` and includes:

- row count;
- selector, join, and planner;
- ASK count and cache hits;
- HTTP request count and transferred bytes;
- parse, source-selection, planning, execution, and total seconds;
- partial-execution status and failed endpoints;
- triple execution order.

## Build and use predicate summaries

The summary path is an experimental metadata mode. It queries each endpoint for
predicate cardinalities, writes a versioned JSON catalog, and can then use that
catalog for source selection and cost ordering.

Build a summary:

```bash
cd go-engine
./fedshop-go summarize \
  --config target/config/config_batch0.ttl \
  --output target/summary/summary_batch0.json \
  --max-concurrency 8 \
  --timeout 120s
```

Run with the summary selector:

```bash
cd go-engine
./fedshop-go query \
  --config target/config/config_batch0.ttl \
  --query ../fedshop-py/benchmark/generation/q01/instance_0/injected.sparql \
  --out-result /tmp/fedshop-go-results.csv \
  --out-source-selection /tmp/fedshop-go-source-selection.csv \
  --query-plan /tmp/fedshop-go-plan.txt \
  --stats /tmp/fedshop-go-stats.json \
  --selector summary \
  --summary target/summary/summary_batch0.json \
  --planner cost \
  --join bind \
  --exclusive-groups \
  --timeout 120s
```

Summary construction cost is stored inside the summary JSON and should be
reported separately from per-query execution cost.

## Run through fedshop-py

`fedshop-py` is the benchmark pipeline wrapper. Its `fedshop-go` adapter:

- builds `go-engine/fedshop-go`;
- generates `go-engine/target/config/config_batch{N}.ttl`;
- invokes `fedshop-go query`;
- transforms raw outputs into canonical `results.csv` and `provenance.csv`;
- writes standard FedShop `stats.csv`.

Install or sync the Python environment:

```bash
cd fedshop-py
uv sync
```

Compile the Go engine through the adapter:

```bash
cd fedshop-py
uv run fedshop evaluate prerequisites fedshop-go --config config/config_small.yaml
```

Generate an engine config for one batch:

```bash
cd fedshop-py
uv run fedshop evaluate generate-config fedshop-go 0 --config config/config_small.yaml
```

Run one benchmark cell:

```bash
cd fedshop-py
uv run fedshop evaluate run fedshop-go q01 0 0 0 --config config/config_small.yaml
```

Run all generated queries for `fedshop-go`:

```bash
cd fedshop-py
uv run fedshop evaluate run-all \
  --config config/config_small.yaml \
  --engine fedshop-go
```

Restrict to one query:

```bash
cd fedshop-py
uv run fedshop evaluate run-all \
  --config config/config_small.yaml \
  --engine fedshop-go \
  --query q01
```

## Prepare benchmark inputs in fedshop-py

A normal local benchmark needs data, Virtuoso ingestion, generated queries, and
then evaluation.

Generate product and source data:

```bash
cd fedshop-py
uv run fedshop generate products --config config/config_small.yaml
uv run fedshop generate sources --config config/config_small.yaml
```

Ingest batch 0 into Virtuoso and write the proxy mapping:

```bash
cd fedshop-py
uv run fedshop ingest batch 0 --config config/config_small.yaml
```

Generate injected query instances:

```bash
cd fedshop-py
uv run fedshop query run-all --config config/config_small.yaml --batch-id 0
```

Then run evaluation:

```bash
cd fedshop-py
uv run fedshop evaluate run fedshop-go q01 0 0 0 --config config/config_small.yaml
```

The default `config/config_small.yaml` uses two batches and two query instances.
Full FedShop-scale runs should use Docker for service isolation and longer
timeouts.

## fedshop-go settings in config_small.yaml

The default adapter entry is:

```yaml
evaluation:
  timeout: 120
  engines:
    fedshop-go:
      dir: "${config_dir}/../../go-engine"
      selector: ask
      cache: memory
      join: bind
      planner: source-count
      exclusive_groups: true
      failure_policy: strict
      max_concurrency: 4
      bind_batch_size: 20
      retry_count: 2
```

Change these values to alter the flags passed by `fedshop-py`. For example,
to use summaries:

```yaml
evaluation:
  engines:
    fedshop-go:
      selector: summary
      planner: cost
```

Before running summary mode through the adapter, build the summary for each
batch:

```bash
cd fedshop-py
uv run fedshop evaluate build-summary 0 --config config/config_small.yaml
uv run fedshop evaluate build-summary 1 --config config/config_small.yaml
```

## Benchmark output locations

A single run writes:

```text
fedshop-py/benchmark/evaluation/fedshop-go/q01/instance_0/batch_0/attempt_0/
  results.txt             # raw engine results from fedshop-go
  results.csv             # normalized result CSV
  source_selection.txt    # raw source-selection CSV from fedshop-go
  provenance.csv          # normalized FedShop provenance matrix
  query_plan.txt          # readable plan
  fedshop_go_stats.json   # engine-owned detailed stats
  stats.csv               # benchmark-standard stats row
```

Generated query inputs live under:

```text
fedshop-py/benchmark/generation/q01/instance_0/
  injected.sparql
  composition.json
  results-batch0.csv
```

Engine configs and summaries live under:

```text
go-engine/target/config/config_batch0.ttl
go-engine/target/summary/summary_batch0.json
```

## Compute metrics

After evaluation runs have written `provenance.csv` files:

```bash
cd fedshop-py
uv run fedshop metrics compute benchmark/metrics.csv --config config/config_small.yaml
```

If no provenance files are passed explicitly, the command scans
`benchmark/evaluation/**/provenance.csv`.

## Known limitations

- `fedshop-go` is scoped to the FedShop query templates, not arbitrary SPARQL.
- The ASK cache is process-local and query-local. It is cold at the start of
  every engine invocation.
- Summary mode only stores predicate cardinalities. It does not estimate full
  join trees, transfer cost, or bound-variable selectivity.
- SELECT execution is simpler than mature engines such as FedX, CostFed, and
  Semagrow. It does not yet implement a full worker scheduler or adaptive
  operators.
- Some query shapes can still materialize very large intermediate result sets.
  Current notes identify q02 as an expected timeout with a very large result
  and q05 as sensitive to independent numeric property cross-products.

## Troubleshooting

### `parse endpoint config: no sd:endpoint services found`

The config file is missing FedShop-style `sd:Service` entries. Generate it
through the adapter:

```bash
cd fedshop-py
uv run fedshop evaluate generate-config fedshop-go 0 --config config/config_small.yaml
```

### Every triple selects every endpoint

Check endpoint graph isolation. Each endpoint should include the correct graph,
usually through `default-graph-uri`, or should otherwise be backed by a
Virtuoso endpoint registered for only that named graph.

### Runtime timeout

Increase the benchmark timeout in `fedshop-py/config/config_small.yaml`:

```yaml
evaluation:
  timeout: 300
```

For standalone runs, pass a larger Go duration:

```bash
--timeout 300s
```

### Adapter reports `error_runtime`

Run the same command manually from `query_plan.txt` and
`fedshop_go_stats.json` context, or reproduce with the direct `fedshop-go
query` command. The adapter records `error_runtime` when the process exits
non-zero.

### `fedshop-py` run writes empty artifacts

Check `stats.csv`. Empty artifacts are expected for timeout propagation,
runtime failure, missing generated query inputs, or explicit `--noexec` paths.

### Summary mode fails with `--planner cost requires --selector summary and --summary`

Use `--selector summary --summary <path> --planner cost` together, or set both
`selector: summary` and `planner: cost` in the `fedshop-go` config entry and
build the summary first.

## Development checks

Run Go tests:

```bash
cd go-engine
GOCACHE=$PWD/.gocache go test ./...
```

Run the Python adapter tests:

```bash
cd fedshop-py
uv run pytest tests/test_engines/test_fedshop_go.py
```

Run all Python tests:

```bash
cd fedshop-py
uv run pytest
```

Before claiming a benchmark result, verify the concrete artifact paths:

- `results.csv` matches the reference result shape for the query.
- `source_selection.txt` has one row per triple pattern.
- `provenance.csv` has `tp*` columns derived from `composition.json`.
- `stats.csv` has numeric values instead of `timeout` or `error_runtime`.
- `fedshop_go_stats.json` has plausible request and timing counts.
