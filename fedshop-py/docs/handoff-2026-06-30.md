## Mudanças nos adapters

### `src/fedshop/engines/pyfedx.py`

`_write_stats` agora inclui `join_time`:
```python
exec_s = _v("execution_seconds")  # só presente no JSON do go
join_s = exec_s if exec_s is not None else max(0.0, total_s - ss_s - plan_s)
row["join_time"] = failed_reason if failed_reason else join_s
```
Isso funciona para **fedshop-go** (usa `execution_seconds` do JSON) e para **pyfedx** (calcula por diferença).

### `src/fedshop/engines/fedx.py`

`_write_stats` agora:
1. Divide `source_selection_time` e `planning_time` por 1000 (ms → s — eram gravados em ms pelo Java)
2. Calcula `join_time = exec_time - source_selection_time - planning_time`

### `src/fedshop/engines/rsa.py`

`run_benchmark` agora:
1. Detecta Java 11+ em `/opt/homebrew/opt/openjdk@{26,21,17,11}/bin/java` se JAVA_HOME não estiver definido
2. Mede `fedup_seconds` (FedUP = source assignment + planning) separadamente de `arq_seconds` (join execution)
3. Escreve `source_selection_time.txt`, `planning_time.txt`, `join_time.txt`

`_write_stats` inclui `join_time` na lista de métricas.

### `src/fedshop/engines/splendid.py`

`_write_stats` inclui `join_time` na lista (vai retornar NaN pois `join_time.txt` não existe — correto).

## Processo de re-coleta de timing do RSA

O RSA precisou de runs adicionais porque:
1. `attempt_0` (run original): sem timing separado de FedUP vs ARQ
2. `attempt_1`: falhou (Java 8 no ambiente, FedUP requer Java 21+)
3. `attempt_2`: paralelo, alguns batch_1 deram timeout por concorrência no TDB2
4. `attempt_4`: runs sequenciais para os casos faltantes de batch_1

O adapter do RSA agora detecta Java 21+ automaticamente. Futuros runs produzirão timing correto diretamente.

**Bug de concorrência TDB2**: quando múltiplos processos FedUP leem o mesmo diretório TDB2 simultaneamente, podem ocorrer timeouts. Rodar RSA com `--max-concurrency 1` por batch resolve. O `run_rsa_attempt2.sh` usou paralelismo de 4 processos.

## Como regenerar tudo do zero

```bash
cd fedshop-py

# 1. Rodar benchmark de todas as engines (attempt_0)
uv run fedshop evaluate run-all --bench-dir benchmark/rerun-$(date +%Y-%m-%d) --engine <engine>

# 2. Patchear timing retroativo (fedshop-go, pyfedx, fedx)
uv run python scripts/patch_stats_join_time.py benchmark/rerun-$(date +%Y-%m-%d)

# 3. Rodar RSA adicional para timing (attempts 2 e 4, sequencial para batch separados)
# Ver scripts/run_rsa_attempt2.sh como referência

# 4. Patchear timing do RSA
uv run python scripts/patch_rsa_timing.py benchmark/rerun-$(date +%Y-%m-%d)

# 5. Gerar metrics_primary.csv (apenas attempt_0 para RSA)
uv run python3 -c "
import pandas as pd
df = pd.read_csv('benchmark/rerun-$(date +%Y-%m-%d)/metrics.csv')
primary = pd.concat([df[df['engine'] != 'rsa'], df[(df['engine']=='rsa') & (df['attempt']==0)]])
primary.to_csv('benchmark/rerun-$(date +%Y-%m-%d)/metrics_primary.csv', index=False)
"

# 6. Gerar tabelas Typst
uv run python scripts/render_typst_tables.py \
    benchmark/rerun-$(date +%Y-%m-%d)/metrics_primary.csv \
    --output benchmark/rerun-$(date +%Y-%m-%d)/typst-tables.typ

# 7. Computar metrics completo
uv run fedshop metrics compute \
    --config config/config_small.yaml \
    --bench-dir benchmark/rerun-$(date +%Y-%m-%d) \
    benchmark/rerun-$(date +%Y-%m-%d)/metrics.csv
```

## Erros conhecidos / pontos de atenção

### FedX: bug de unidade ms→s (corrigido)
`source_selection_time` e `planning_time` eram gravados em ms pelo Java (e.g. `113`), mas o resto das engines usa segundos. O patch retroativo corrige os stats.csv existentes e o adapter futuro também converte.

### RSA: requer Java 21+
O jar FedUP foi compilado com Java 26. O adapter agora detecta `/opt/homebrew/opt/openjdk@{26,21,17,11}/bin/java` automaticamente. Se rodar em outro ambiente, defina `JAVA_HOME` ou adicione `java_bin` na config:
```yaml
rsa:
  java_bin: /path/to/java21/bin/java
```

### Semagrow e SPLENDID: sem timing interno
São caixas pretas Java que não expõem breakdowns de fase. `join_time`, `planning_time`, `source_selection_time` permanecerão `--` nas tabelas. Para obter esses dados seria necessário instrumentar o código Java dessas engines.

### Concorrência TDB2 no RSA
O FedUP usa TDB2 para o índice de seleção de fontes. Múltiplos processos FedUP no mesmo diretório TDB2 podem competir e causar timeouts. Rodar instâncias de batch diferentes em paralelo é seguro (usam diretórios distintos), mas instâncias diferentes do mesmo batch competem pelo mesmo TDB2.

## Scripts de tabela Typst

`scripts/render_typst_tables.py` gera 3 tabelas do `metrics_primary.csv`:
1. `<res:tempo-engines>` — Timing detalhado (exec, planning, source_selection, join) por engine/categoria
2. `<tab:tempo>` — Sumário (tempo médio, taxa de sucesso, método de seleção)
3. `<tab:corretude>` — Corretude em queries com resultados não-nulos (q06-q12): precisão, revocação, F1

Modo de uso:
```bash
uv run python scripts/render_typst_tables.py <metrics.csv> [-o output.typ] [--mode all|timing|summary|correctness]
```
