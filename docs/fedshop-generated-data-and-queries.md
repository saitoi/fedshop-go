# FedShop Generated Data And Queries

This guide explains the shape of the data and query workload generated by
FedShop. It is meant to help when implementing or debugging a federated engine:
what kinds of triples will appear, which predicates are common, and what query
patterns the engine must handle.

The local generated dataset directory is not populated by default in this repo,
so this guide is based on the generator templates, config, wiki, and paper:

- `reference-repos/FedShop/experiments/bsbm/snakefile/config.yaml`
- `reference-repos/FedShop/experiments/bsbm/model/watdiv/bsbm-product.template`
- `reference-repos/FedShop/experiments/bsbm/model/watdiv/bsbm-vendor.template`
- `reference-repos/FedShop/experiments/bsbm/model/watdiv/bsbm-ratingsite.template`
- `reference-repos/FedShop/experiments/bsbm/queries/q*.sparql`
- `reference-papers/fedshop.pdf`

## Dataset Model

FedShop models a federated e-commerce site.

There are three conceptual data zones:

```text
virtual catalog
autonomous vendors
autonomous rating sites
```

The virtual catalog defines the global products, product types, product
features, and producers. It is generated first, but it is not used as one
central endpoint in the final federation.

Instead, vendors and rating sites replicate product/catalog data using local
URIs. Local entities are connected back to global catalog entities with
`owl:sameAs`.

The important pattern is:

```text
global product/type/feature/producer
local product/type/feature/producer in endpoint A
local product/type/feature/producer in endpoint B
owl:sameAs links from local entities to global entities
```

This is why many FedShop queries contain `owl:sameAs`: the query asks for a
global product/type/feature, but each endpoint stores local copies.

## Federation Growth

The full config generates 10 batches.

Each batch adds:

```text
10 vendors
10 rating sites
```

So the benchmark grows like:

```text
batch 0 -> 20 endpoints
batch 1 -> 40 endpoints
...
batch 9 -> 200 endpoints
```

The full config uses:

```yaml
product_n: 200000
vendor_n: 10 * n_batch
ratingsite_n: 10 * n_batch
```

At the final scale, the federation has:

```text
100 vendor endpoints
100 rating-site endpoints
```

FedShop does not normally start one physical database per endpoint. It loads the
federation into Virtuoso and exposes many virtual endpoints through the FedShop

For an engine, this still looks like many SPARQL endpoints.

## Virtual Catalog Data

The product template generates these main entity types:

```text
bsbm:Product
bsbm:ProductType
bsbm:ProductFeature
bsbm:Producer
```

### Product Triples

Products are the densest catalog entities.

Common product predicates:

```text
rdf:type
rdfs:label
rdfs:comment
bsbm:producer
bsbm:productFeature
bsbm:productPropertyTextual1
bsbm:productPropertyTextual2
bsbm:productPropertyTextual3
bsbm:productPropertyTextual4
bsbm:productPropertyTextual5
bsbm:productPropertyNumeric1
bsbm:productPropertyNumeric2
bsbm:productPropertyNumeric3
bsbm:productPropertyNumeric4
bsbm:productPropertyNumeric5
bsbm:publishDate
```

Always-present product-ish properties in the template:

```text
rdfs:label
rdfs:comment
bsbm:productPropertyTextual1
bsbm:productPropertyTextual2
bsbm:productPropertyTextual3
bsbm:productPropertyNumeric1
bsbm:productPropertyNumeric2
bsbm:productPropertyNumeric3
bsbm:publishDate
```

Optional product properties:

```text
bsbm:productPropertyTextual4  probability 0.7
bsbm:productPropertyTextual5  probability 0.8
bsbm:productPropertyNumeric4  probability 0.7
bsbm:productPropertyNumeric5  probability 0.8
```

Associations:

```text
Product -> rdf:type -> ProductType
Product -> bsbm:productFeature -> ProductFeature
Product -> bsbm:producer -> Producer
```

The config sets `type_c: 9` and `feature_c: 9`, so product type and product
feature links are very common. These are important for source selection because
many product-search queries filter through type and feature joins.

### ProductType, ProductFeature, Producer

These are descriptive catalog entities.

Common predicates:

```text
rdfs:label
rdfs:comment
bsbm:publishDate
```

Producer also has:

```text
bsbm:country
```

## Vendor Endpoint Data

Each vendor endpoint represents one shop.

Main local entity types:

```text
bsbm:Vendor
bsbm:Offer
local bsbm:Product replicas
```

### Vendor Triples

Each vendor has:

```text
rdfs:label
rdfs:comment
bsbm:country
foaf:homepage
bsbm:publishDate
```

### Offer Triples

Offers are the main vendor-specific facts.

Common offer predicates:

```text
bsbm:product
bsbm:vendor
bsbm:price
bsbm:validFrom
bsbm:validTo
bsbm:deliveryDays
bsbm:offerWebpage
bsbm:publishDate
```

Associations:

```text
Offer -> bsbm:vendor -> Vendor
Offer -> bsbm:product -> Product
```

The config uses an average-ish offer count of about 2000 per vendor template
instance. The exact generated count is distribution-driven, not a fixed constant
per endpoint.

### Local Product Replicas

Vendor data depends on the catalog product fragments.

When products are replicated into a vendor endpoint:

```text
global product URI is copied into a local URI space
local product keeps product facts needed by queries
local product/type/feature links back to global catalog with owl:sameAs
```

The important engine consequence:

```text
The same global product can appear in many vendor endpoints under different
local URIs.
```

So a source-selection algorithm cannot assume a product URI appears only once.

## Rating Site Endpoint Data

Each rating site endpoint represents one review site.

Main local entity types:


### RatingSite Triples

Each rating site has:

Common review predicates:

```text

bsbm:rating3

bsbm:publishDate
bsbm:reviewDate
bsbm:reviewFor
```

Always-present review-ish properties:

```text
dc:title

bsbm:publishDate
```

```text
bsbm:rating1 probability 0.7
```

### Person Triples

Common person predicates:

```text

bsbm:country
bsbm:publishDate
```

## Most Common Predicate Families

Without generated local data files, we should avoid claiming exact measured
frequencies. From the templates, the most common predicate families are clear.

Very common in product/catalog replicas:

```text
owl:sameAs
rdf:type
rdfs:label
bsbm:productFeature
bsbm:productPropertyTextual*
bsbm:productPropertyNumeric*
bsbm:producer
```

Very common in vendor endpoints:

```text
bsbm:product
bsbm:vendor
bsbm:price
bsbm:validTo
bsbm:deliveryDays
bsbm:offerWebpage
```

Very common in rating-site endpoints:

```text
bsbm:reviewFor
rev:reviewer
dc:title
rev:text
bsbm:reviewDate
bsbm:rating*
```

Common metadata predicates across many entity types:

```text
rdfs:label
rdfs:comment
bsbm:country
bsbm:publishDate
```

For source selection, the most important predicates are often not just the most
frequent ones, but the ones that distinguish endpoint type:

```text
vendor-ish:      bsbm:offerWebpage, bsbm:price, bsbm:deliveryDays
rating-site-ish: rev:text, rev:reviewer, bsbm:reviewFor, bsbm:reviewDate
catalog-ish:     bsbm:productFeature, rdf:type, owl:sameAs
```

## Query Workload

FedShop adapts the BSBM Explore query workload.

The full workload is:

```text
12 query templates
10 instantiated queries per template
120 total query instances
```

The templates are:

```text
q01 through q12
```

Each template has a `.sparql` file and a `.const.json` file:

```text
reference-repos/FedShop/experiments/bsbm/queries/q01.sparql
reference-repos/FedShop/experiments/bsbm/queries/q01.const.json
```

The `.sparql` files contain variables/placeholders. Query generation injects
real values and writes instantiated queries under the benchmark generation
directory.

## Query Artifacts

For each generated query instance, FedShop produces files like:

```text
injected.sparql
composition.json
provenance.sparql
provenance.csv
results.csv
```

Meaning:

- `injected.sparql`: the actual SPARQL query with constants filled in.
- `composition.json`: mapping of triple-pattern names like `tp1` to actual
  triple patterns.
- `provenance.sparql`: query used to compute source/provenance information.
- `provenance.csv`: reference source assignment data.
- `results.csv`: expected centralized result.

The provenance files are important for RSA, but normal engines like FedX receive
the raw query and must do source selection themselves.

## Query Shapes

The paper characterizes the 12 templates roughly like this:

| Query | Triple patterns | Main shape |
| --- | ---: | --- |
| q01 | 8 | product search by type/features/numeric filter |
| q02 | 16 | product detail lookup |
| q03 | 11 | product search with optional exclusion |
| q04 | 18 | product search with `UNION` |
| q05 | 11 | similar-product query |
| q06 | 2 | label regex search |
| q07 | 17 | product + offers + reviews, optional-heavy |
| q08 | 11 | reviews for a product |
| q09 | 1 | reviewer lookup |
| q10 | 7 | offer lookup with price/delivery filters |
| q11 | 3 | offer/product property exploration with `UNION` |
| q12 | 10 | offer detail lookup |

Common SPARQL features in the workload:

```text
SELECT
SELECT DISTINCT
FILTER
OPTIONAL
UNION
ORDER BY
LIMIT
regex
langMatches
bound / !bound
numeric comparisons
date comparisons
```

This matters for a Go implementation: a parser that only supports plain BGPs
will not run the full workload.

## Product Queries

Several queries are product-centric.

They tend to use:

```text
rdfs:label
rdf:type
bsbm:productFeature
bsbm:productPropertyNumeric*
bsbm:productPropertyTextual*
owl:sameAs
```

The common source-selection challenge:

```text
Many endpoints may contain local replicas of product/catalog data.
```

So product-only triple patterns can have many candidate sources.

## Offer Queries

Offer queries are vendor-centric.

They tend to use:

```text
bsbm:product
bsbm:vendor
bsbm:price
bsbm:deliveryDays
bsbm:validTo
bsbm:offerWebpage
foaf:homepage
```

The common source-selection challenge:

```text
Offer predicates should mostly select vendor endpoints, not rating-site
endpoints.
```

This is useful for a simple engine: predicate-family summaries can immediately
prune rating sites for offer-only patterns.

## Review Queries

Review queries are rating-site-centric.

They tend to use:

```text
bsbm:reviewFor
rev:reviewer
dc:title
rev:text
bsbm:reviewDate
bsbm:rating1
bsbm:rating2
bsbm:rating3
bsbm:rating4
foaf:name
```

The common source-selection challenge:

```text
Review predicates should mostly select rating-site endpoints, not vendor
endpoints.
```

## Cross-Domain Queries

Some queries combine product, offer, and review information.

The hard case is:

```text
same global product
-> local product in vendor endpoint
-> local product in rating-site endpoint
-> join through owl:sameAs/global product identity
```

This is where naive source selection and decomposition often become expensive.

If the engine sends too many small subqueries to too many endpoints, the number
of HTTP requests and intermediate rows grows quickly.

## What A Minimal Engine Should Exploit

For a benchmark-specific engine, simple predicate-family knowledge is valuable.

A first source-selection index can be:

```text
predicate -> endpoint type
```

Example:

```text
bsbm:price        -> vendors
bsbm:deliveryDays -> vendors
bsbm:offerWebpage -> vendors
rev:text          -> rating sites
rev:reviewer      -> rating sites
bsbm:reviewFor    -> rating sites
```

Then use endpoint-specific ASK checks only after this coarse pruning.

Better:

```text
predicate -> concrete endpoints that contain predicate
```

Better still:

```text
predicate + bound object/type/feature -> concrete endpoints
```

CostFed-like behavior starts when these summaries also include cardinality
estimates and are used to choose join order.

## Implementation Consequences

A FedShop-focused Go engine should represent at least:

```go
type EntityClass string

const (
    VendorEndpoint EntityClass = "vendor"
    RatingEndpoint EntityClass = "ratingsite"
)

type PredicateSummary struct {
    Predicate string
    Endpoints []string
    Count     int64
}

type EndpointSummary struct {
    EndpointID string
    Kind       EntityClass
    Predicates map[string]PredicateSummary
}
```

This is enough to implement a useful first pass:

```text
extract triple patterns
look at predicate
use summary to choose candidate endpoints
run fewer ASK checks
execute selected subqueries
join results
```

For FedShop, that already targets the biggest source-selection bottleneck:
avoiding pointless checks against endpoints that are structurally the wrong kind
of source.
