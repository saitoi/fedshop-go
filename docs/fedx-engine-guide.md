# FedX Engine Guide

This guide is for understanding a real federated SPARQL engine before trying to
build one. In the local FedShop forks, FedX is the smallest useful starting
point.

FedX is not small because federation is simple. It is small here because this
fork delegates most of the heavy work to RDF4J: SPARQL parsing, query algebra,
repository connections, query execution, and much of the federation runtime.

## Why Start With FedX

Approximate local source sizes:

| Engine | Source files | LOC | Notes |
| --- | ---: | ---: | --- |
| FedX | 8 | 2.3k | Smallest actual engine fork |
| fedup | 63 | 7.7k | Research prototype style |
| ANAPSID | 55 | 11.8k | Python 2 era environment |
| SPLENDID | 97 | 12.8k | Java, metadata-heavy |
| MULDER | 62 | 12.7k | Python mediator stack |
| Semagrow | 391 | 29.6k | Larger RDF4J-style system |
| Odyssey | 150 | 36.2k | Larger, bundled dependencies |
| CostFed | 276 | 57.1k | Largest; summaries, cost model, FedX lineage |

CostFed is the engine to study after FedX. It adds summaries and cost-aware
planning, but you will understand it faster if you first understand FedX's basic
runtime source selection.

## Main Files

Read these in order.

1. `reference-repos/FedShop/fedshop/engines/fedx.py`

   This is FedShop's Python wrapper for FedX. It is how the benchmark calls the
   engine.

   Important functions:

   - `prerequisites`: builds the FedX Maven project.
   - `generate_config_file`: writes the endpoint config for one FedShop batch.
   - `run_benchmark`: prepares output paths and runs one query.
   - `exec_fedx`: launches the Java entrypoint with Maven.
   - `transform_results`: converts FedX's binding output into CSV.
   - `transform_provenance`: converts FedX source-selection output into the
     FedShop provenance table format.

2. `reference-repos/fedshop-forks/FedX/src/main/java/org/example/FedX.java`

   This is the Java executable that FedShop launches.

   It:

   - reads CLI args from `fedx.py`;
   - creates an RDF4J `FedXRepository`;
   - loads federation members from the generated config;
   - reads the raw SPARQL query;
   - calls `conn.prepareTupleQuery(rawQuery)`;
   - calls `tq.evaluate()`;
   - writes result, query plan, source selection, and timing files.

3. `reference-repos/fedshop-forks/FedX/src/main/java/org/example/FedXContainer.java`

   This is mostly FedShop instrumentation. It stores data that FedShop wants to
   report:

   - selected sources per triple pattern;
   - source-selection time;
   - planning time;
   - ASK count;
   - optional forced source-selection data.

4. `reference-repos/fedshop-forks/FedX/src/main/java/org/eclipse/rdf4j/federated/optimizer/SourceSelection.java`

   This is the most important file. It decides which endpoints can answer each
   triple pattern.

5. `reference-repos/fedshop-forks/FedX/src/main/java/org/eclipse/rdf4j/federated/evaluation/SparqlTripleSource.java`

   This shows how FedX checks an endpoint, usually with a remote SPARQL `ASK`.

## High-Level Flow

FedShop starts the engine like this:

```text
fedshop/engines/fedx.py
  -> generate_config_file(...)
  -> run_benchmark(...)
  -> exec_fedx(...)
  -> mvn exec:java -Dexec.mainClass="org.example.FedX"
```

Then Java takes over:

```text
FedX.main(...)
  -> FedXFactory.newFederation()
  -> withMembers(config_batchN.ttl)
  -> repo.getConnection()
  -> conn.prepareTupleQuery(rawQuery)
  -> tq.evaluate()
```

The call to `tq.evaluate()` is where RDF4J/FedX parses, optimizes, performs
source selection, plans, sends endpoint requests, and joins results.

## What FedShop Passes To FedX

Normal FedX receives the raw FedShop query, not a pre-filled RSA `SERVICE`
query.

```text
raw SPARQL query
+ endpoint config for the current batch
+ timeout
+ output paths
+ noexec flag
```

The endpoint config is generated by `generate_config_file` as a Turtle file like:

```turtle
@prefix sd: <http://www.w3.org/ns/sparql-service-description#> .
@prefix fedx: <http://rdf4j.org/config/federation#> .

<http://www.vendor.example/> a sd:Service ;
    fedx:store "SPARQLEndpoint";
    sd:endpoint "http://proxy/sparql_www.vendor.example_";
    fedx:supportsASKQueries true .
```

This tells FedX which federation members exist for the batch and which SPARQL
URL to use for each member.

## Source Selection

The key entrypoint is:

```java
doSourceSelection(List<StatementPattern> stmts)
```

It dispatches to one of two modes:

```java
doSourceSelectionDefault(stmts)
doSourceSelectionForce(stmts)
```

Default mode is the real FedX behavior. Forced mode is FedShop-specific
instrumentation used when source assignments are supplied externally.

In default mode, the logic is:

```text
for each triple pattern:
  for each endpoint:
    check the source-selection cache
    if cache says yes:
      record endpoint as a source
    if cache says no:
      skip endpoint
    if cache is unsure:
      send a remote ASK query
      cache the answer
      if ASK is true:
        record endpoint as a source
```

After sources are selected, FedX rewrites the query algebra:

```text
0 sources -> EmptyStatementPattern
1 source  -> ExclusiveStatement
N sources -> StatementSourcePattern
```

That rewrite is the bridge between source selection and execution. The rest of
the FedX/RDF4J runtime uses those algebra nodes to decide where subqueries go.

## Endpoint ASK Checks

`SparqlTripleSource.hasStatements(...)` performs the endpoint check.

If ASK queries are enabled, it builds an `ASK` query for the triple pattern and
sends it to the endpoint:

```text
ASK WHERE {
  triple pattern
}
```

If the endpoint returns true, that endpoint is considered relevant for that
triple pattern.

This is simple and accurate, but expensive at scale:

```text
number of ASK requests ~= triple patterns * endpoints
```

For FedShop with hundreds of endpoints, that can become a serious bottleneck.
CostFed exists largely to improve this by using summaries and cost estimates
instead of blindly probing every endpoint.

## Example Mental Model

For a query like:

```sparql
SELECT ?product ?label WHERE {
  ?product a ?type .
  ?product rdfs:label ?label .
}
```

FedX sees two statement patterns:

```text
tp1: ?product a ?type
tp2: ?product rdfs:label ?label
```

It may discover:

```text
tp1 -> endpoint A, endpoint B
tp2 -> endpoint B, endpoint C
```

Then the execution layer can query the selected endpoints and join bindings on
shared variables such as `?product`.

## What To Ignore First

To understand the engine, ignore these initially:

- CSV result formatting in `transform_results`;
- FedShop provenance reshaping in `transform_provenance`;
- Maven build details;
- RDF4J internals outside the files listed above;
- forced source-selection mode;
- monitoring and logging;
- advanced join algorithms.

Focus on this path:

```text
FedShop wrapper
-> Java main
-> source selection
-> endpoint ASK
-> algebra rewrite
```

## FedX Versus RSA

FedX and RSA are very different.

FedX:

```text
raw query + endpoint list
-> engine decides sources at runtime
-> engine executes
```

RSA:

```text
raw query + precomputed provenance/source assignment
-> FedShop creates SERVICE query
-> Jena executes the already-selected sources
```

RSA is a reference baseline with privileged knowledge. FedX is a real engine
that has to discover relevant sources during execution.

## FedX Versus CostFed

FedX asks endpoints directly:

```text
Does endpoint E have data for triple pattern TP?
```

CostFed tries to avoid excessive endpoint probing by using summaries and cost
estimates. Conceptually, CostFed adds:

- source summaries;
- cardinality estimates;
- source ranking;
- join-order optimization;
- more planner logic.

That is why CostFed is harder to read first. It is not just "FedX but slightly
bigger"; it adds a layer of statistics and planning on top of the basic
federation model.

## Study Checklist

Use this order:

1. Read `fedx.py` and identify every argument passed to Java.
2. Read `FedX.java` and find where the query is prepared and evaluated.
3. Read `FedXContainer.java` to understand what data is captured for FedShop.
4. Read `SourceSelection.java` line by line.
5. Trace one triple pattern through `doSourceSelectionDefault`.
6. Read `SparqlTripleSource.hasStatements`.
7. Draw how one query turns into endpoint ASK checks and selected sources.
8. Only then move to CostFed's source summaries and cost model.

## Mapping This To A Go Engine

A minimal Go engine modeled after FedX for FedShop would need:

```text
parse FedShop SPARQL subset
extract triple patterns
load endpoint list for batch
ASK each endpoint for each triple pattern
build source-selection table
decompose query into endpoint subqueries
execute subqueries
join bindings
write FedShop-compatible outputs
```

This would not be CostFed. It would be a FedX-like baseline. After that works,
the CostFed-like improvement is:

```text
replace ASK-every-endpoint source selection
with summary-based source selection and cost-aware planning
```

That is the right progression:

```text
RSA-style executor
-> FedX-like ASK-based engine
-> CostFed-like summary/cost-based engine
```
